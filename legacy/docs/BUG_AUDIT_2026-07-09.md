## P0 - Unauthenticated write/read/compute endpoints are exposed

### 1. Unauthenticated `PUT /api/cameras/{camera_id}/zones` can modify server-side config files

- Evidence in code:
  - [tf_api/api/routes_zones.py](/home/ttung05/Desktop/SU26/vehicle_counting/tf_api/api/routes_zones.py:86)
  - [tf_api/api/routes_zones.py](/home/ttung05/Desktop/SU26/vehicle_counting/tf_api/api/routes_zones.py:108)
- Problem:
  - both `GET` and `PUT` zone endpoints have no `Depends(get_current_user)` or `Depends(require_operator)`
  - unauthenticated clients can rewrite `configs/detection_zones/{camera_id}_zones.yaml`
- Runtime proof:
  - anonymous `PUT http://127.0.0.1:8014/api/cameras/YT_LIVE_TEST/zones` returned `200`
- Impact:
  - arbitrary ROI reconfiguration
  - can break inference geometry for all users
  - config tampering without login
- Fix:
  - require at least JWT for `GET`
  - require operator/admin for `PUT`
  - add regression tests for anonymous `401/403`

## P1 - Data correctness bugs

### 3. `/api/cameras/{camera_id}/counts/recent` returns lane-change-style rows instead of recent count events

- Evidence in code:
  - [tf_api/api/routes_counts.py](/home/ttung05/Desktop/SU26/vehicle_counting/tf_api/api/routes_counts.py:108)
  - [tf_api/api/routes_counts.py](/home/ttung05/Desktop/SU26/vehicle_counting/tf_api/api/routes_counts.py:119)
  - [tf_db/repositories.py](/home/ttung05/Desktop/SU26/vehicle_counting/tf_db/repositories.py:234)
- Problem:
  - endpoint doc says "recent crossing events"
  - implementation calls `repo.get_lane_changes(...)`
  - repository method filters only rows with non-empty `direction`, and returns a reduced lane-change-like payload
- Impact:
  - frontend "recent counts" can silently miss valid count events
  - semantics of the endpoint do not match its name or docs
- Fix:
  - add a dedicated `get_recent_events()` query over `VehicleCountEvent`
  - return the full count-event payload
  - keep lane-change reporting in its own endpoint only

### 4. Latest occupancy calculation is wrong: code comments say unique tracks, implementation counts grouped rows only

- Evidence in code:
  - [tf_db/repositories.py](/home/ttung05/Desktop/SU26/vehicle_counting/tf_db/repositories.py:328)
  - [tf_db/repositories.py](/home/ttung05/Desktop/SU26/vehicle_counting/tf_db/repositories.py:340)
- Problem:
  - comment says "Count unique track_ids per lane"
  - actual query groups only by `lane_id`
  - loop then increments by 1 per returned lane row, so occupancy tends toward `1` per active lane, not real vehicle occupancy
- Impact:
  - `/api/cameras/{id}/occupancy/latest`
  - reports occupancy field
  - dashboard/system views that depend on latest occupancy
- Fix:
  - use `count(distinct VehicleCountEvent.track_id)` per lane over a bounded freshness window
  - define a clearer occupancy source of truth if event rows are not sufficient

- Evidence in code:

    - [tf_api/api/routes_settings.py](/home/ttung05/Desktop/SU26/vehicle_counting/tf_api/api/routes_settings.py:52)
  - frontend expects nested sections:
    - [frontend/js/pages/settings.js](/home/ttung05/Desktop/SU26/vehicle_counting/frontend/js/pages/settings.js:52)
    - [frontend/js/pages/settings.js](/home/ttung05/Desktop/SU26/vehicle_counting/frontend/js/pages/settings.js:73)
    - [frontend/js/pages/settings.js](/home/ttung05/Desktop/SU26/vehicle_counting/frontend/js/pages/settings.js:83)
    - [frontend/js/pages/settings.js](/home/ttung05/Desktop/SU26/vehicle_counting/frontend/js/pages/settings.js:93)
    - [frontend/js/pages/settings.js](/home/ttung05/Desktop/SU26/vehicle_counting/frontend/js/pages/settings.js:111)
- Runtime proof:
  - `GET /api/settings` currently returns only:
    - `api_url`
    - `max_workers`
    - `output_dir`
    - minimal `detection`
  - it does not return `system`, `storage`, `notifications`, `appearance`
- Impact:
  - settings page shows/edits fields that the backend does not persist
  - users think they changed runtime behavior, but most values are ignored
- Fix:

  - add round-trip tests: GET -> edit -> PUT -> GET

### 6. The current `tf_api` runtime does not consume most values saved by the settings page

- Evidence:
  - search results show `configs/settings.json` is actively consumed in legacy `backend/services/settings_service.py`, but not in the current `tf_api` live path
  - [tf_api/api/routes_live.py](/home/ttung05/Desktop/SU26/vehicle_counting/tf_api/api/routes_live.py:174) builds detector/tracking/counting config from camera YAML, not from the settings file
- Impact:
  - changing many settings in the UI does not affect live inference
  - operators can get a false sense that system limits and detection parameters were applied
- Fix:
  - either wire `tf_api` to a real runtime settings service
  - or remove unsupported settings from the UI until they are implemented

## P1 - Runtime and performance regressions

### 7. MotionDetector regression: `tf_api` lost the `source_type` safeguard that existed in the older backend path

- Evidence in code:
  - current live path:
    - [tf_api/api/routes_live.py](/home/ttung05/Desktop/SU26/vehicle_counting/tf_api/api/routes_live.py:343)
  - current detector has no `source_type` disable path:
    - [tf_worker/detection/motion_detector.py](/home/ttung05/Desktop/SU26/vehicle_counting/tf_worker/detection/motion_detector.py:1)
  - older backend version had explicit live-stream disable logic:
    - [backend/detection/motion_detector.py](/home/ttung05/Desktop/SU26/vehicle_counting/backend/detection/motion_detector.py:1)
    - [backend/api/routes_live.py](/home/ttung05/Desktop/SU26/vehicle_counting/backend/api/routes_live.py:375)
- Problem:
  - YouTube/RTSP streams often have compression noise every frame
  - old code disabled motion gating for these sources
  - current `tf_api` path does not
- Impact:
  - wasted CPU on live streams
  - inconsistent "skip detection on static scene" behavior
  - harder to keep `process_fps` close to `source_fps`
- Fix:
  - restore `source_type` support in `tf_worker/detection/motion_detector.py`
  - pass `source_type` from `tf_api/api/routes_live.py`

### 8. Live processing FPS still lags the source FPS on the tested YouTube stream

- Runtime proof:
  - `GET /live/YT_LIVE_TEST/metrics` reported:
    - `source_fps = 30.0`
    - `process_fps = 24.0`
    - `avg_latency_ms = 888.5`
- Impact:
  - frontend is not yet keeping processing in lockstep with the input rate
  - backlog/latency risk increases on busier scenes
- Fix:
  - add explicit adaptive frame dropping / decimation in live mode
  - expose queue depth and per-stage timing in the UI
  - consider batching or reduced detect cadence for live YouTube sources

## P1 - Developer workflow / startup bugs

### 9. `npm run dev` and `npm run start` bypass the repo virtualenv and fail on a clean machine

- Evidence in code:
  - npm scripts use raw `python3`:
    - [package.json](/home/ttung05/Desktop/SU26/vehicle_counting/package.json:7)
    - [package.json](/home/ttung05/Desktop/SU26/vehicle_counting/package.json:9)

  - bootstrap path uses `.venv` correctly:
    - [start.sh](/home/ttung05/Desktop/SU26/vehicle_counting/start.sh:9)
- Runtime proof on this machine:
  - `python3` could not import `yt_dlp`, `fastapi`, `uvicorn`
- Impact:
  - `npm run dev/start/seed/pipeline` can fail even when `./start.sh` works
  - YouTube camera support is especially fragile because `yt_dlp` is missing from system Python
- Fix:
  - point npm scripts to `.venv/bin/python`
  - or add a wrapper script so every entrypoint shares one bootstrap path

### 10. YouTube support still depends on Python 3.10, which `yt-dlp` already warns is deprecated

- Runtime proof:
  - resolving the 3 approved YouTube URLs emitted:
    - `Deprecated Feature: Support for Python version 3.10 has been deprecated`
- Impact:
  - future breakage risk for the core external integration you now rely on
- Fix:
  - move the project runtime to Python 3.11+
  - pin and re-verify OpenCV / torch / yt-dlp compatibility there

## P2 - Release engineering and maintainability gaps

### 11. There is no automated test suite or CI gate in the current repo

- Evidence:
  - no `tests/` tree found
  - no `.github/workflows/` found
  - no `test-reports/` artifacts found
- Impact:
  - auth regressions, live-stream regressions, and DB correctness bugs can reappear unnoticed
  - there is no release gate for the YouTube/live pipeline path
- Fix:
  - add at minimum:
    - auth tests
    - camera snapshot/live smoke tests
    - DB repository tests
    - settings round-trip tests
  - wire them into CI

### 12. Duplicate code trees (`backend/` and `tf_*`) are still present and can drift apart

- Evidence:
  - [backend/main.py](/home/ttung05/Desktop/SU26/vehicle_counting/backend/main.py:1) is only a compatibility wrapper
  - but full duplicated route trees still exist under both:
    - `backend/api/*.py`
    - `tf_api/api/*.py`
- Impact:
  - fixes can land in the wrong tree
  - old logic can diverge from the live code path
  - this already happened around snapshot/motion-detection behavior
- Fix:
  - remove or archive dead backend duplicates
  - keep a single authoritative implementation path

### 13. Empty accidental brace-named directories are committed and add noise

- Evidence:
  - `tf_api/{api,services,monitoring,config}`
  - `tf_worker/{storage,events,io,visualization,detection,monitoring,evaluation,benchmark}`
  - `tf_core/{config,counting,detection,lanes,occupancy,tracking,schemas,io}`
- Impact:
  - IDE/search noise
  - signals previous automation mistakes in repo layout
- Fix:
  - delete these empty directories
  - add a lightweight repo hygiene check

### 14. Frontend bundle strategy is drift-prone: `app.js` duplicates `core.js + pages/*.js`

- Evidence:
  - AGENTS workflow says production bundle is rebuilt manually by concatenation
  - the repo contains both:
    - `frontend/js/core.js`
    - `frontend/js/pages/*.js`
    - `frontend/js/app.js`
- Impact:
  - easy to fix one file and forget the bundled copy
  - behavior can differ between development assumptions and shipped bundle
- Fix:
  - replace manual concatenation with a real build step
  - or load modular files directly and stop committing a duplicated bundle
